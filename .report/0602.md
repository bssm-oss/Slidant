# Slidant vs Snapdeck — 경쟁 분석 보고서
**작성일**: 2026-06-02  
**비교 대상**: https://www.snapdeck.app/ vs http://localhost/drive

---

## 요약 판정

Snapdeck은 **완성품**이다. Slidant는 **기술 데모**다.  
핵심 루프(입력 → 생성 → 완료)가 Snapdeck은 3단계, Slidant는 현재 불완전하다.  
Agent 협업이라는 차별점이 있지만 UX로 드러나지 않아 경쟁 우위가 없다.

---

## 1. 첫인상 / 진입 경험

### Snapdeck
- 랜딩 페이지 자체가 생성 UI. 텍스트 박스 하나로 즉시 시작
- 슬라이드 수 조절(+/-), Web Research / Web Scrap / Import Notion / Auto Agent 옵션 제공
- 로그인 없이 첫 화면에서 가치 전달 완료

### Slidant
- 드라이브 페이지가 진입점. 생성 버튼("새 프레젠테이션")을 찾아야 함
- 버튼 클릭 → 빈 에디터 열림 → 직접 Agent에게 입력해야 생성 시작
- 입력 소스 확장(Web 검색, URL 스크랩, Notion 연동) **전혀 없음**
- **첫인상 격차: 치명적**

---

## 2. 드라이브 / 대시보드

### Snapdeck
- 카드에 실제 슬라이드 콘텐츠가 썸네일로 보임
- 탭 필터(전체 / 최근 / 공유됨), 검색, 그리드/리스트 토글
- 공유 탭이 협업 진입점 역할

### Slidant — 문제점

| 문제 | 심각도 |
|------|--------|
| 12개 프레젠테이션 중 대부분 "제목 없는 프레젠테이션" | 높음 |
| HTML content 슬라이드 썸네일 모두 "빈 슬라이드"로 표시 | 높음 |
| 공유 기능 없음 | 중간 |
| 정렬/필터 옵션 없음 (필터 버튼은 있으나 기능 미확인) | 낮음 |

**근본 원인**: `SlideListPanel`의 `MiniComponent`와 `SlideThumbnail`이 JSON 컴포넌트 방식만 지원. `html_content` 있는 슬라이드는 iframe 없이 렌더링되므로 썸네일이 항상 빈 슬라이드.  
→ `html_content` 슬라이드는 백엔드에서 썸네일 PNG를 별도 생성하거나, 클라이언트에서 숨겨진 iframe으로 캡처해야 함.

## 3. 에디터 UX

### Snapdeck
- 상단 툴바: 뒤로/앞으로, 아웃라인, 슬라이드 재생성, 이전/다음 슬라이드
- 하단 툴바: Select / Pan / Comment / Text / Image / Icon library / Shape / Magic / Undo / Redo
- 우측: Chat | Design 탭 전환
- Design 탭: 배경색 · 불투명도 즉시 조작
- 재생 버튼(프레젠테이션 모드) · Export PDF · Share 버튼이 항상 노출
- 모델 버전 선택 UI (Snapdeck 1.0)

### Slidant — 문제점

#### 3-1. 없는 기능들 (경쟁사 대비 누락)
- **Export(PDF/PPTX) 버튼 없음** — 완성물을 꺼낼 방법이 없다. 제품의 가장 기본 출구가 막혀있음
- **프레젠테이션 모드(전체 화면 뷰어) 없음** — 결과물을 발표할 수 없음
- **공유/협업 링크 없음** — "협업 플랫폼"인데 공유 불가
- **Undo/Redo 없음** — HTML mode에서는 히스토리 패널 경유 필요. 일반 사용자 멘탈 모델과 완전히 불일치
- **슬라이드 재생성(Regenerate) 버튼 없음** — 특정 슬라이드만 다시 만들려면 채팅으로 요청해야 함
- **캔버스 직접 편집 불가(HTML mode)** — iframe sandbox 때문에 요소 클릭/드래그 불가. JSON 컴포넌트 모드에서만 편집 가능한데, Agent 생성 결과는 HTML mode임. 즉 Agent가 만든 슬라이드는 수동 수정 불가

#### 3-2. 있지만 깨진 것들
- **슬라이드 패널 썸네일**: HTML content 슬라이드는 좌측 패널에서도 "빈 슬라이드" 표시
- **제목 자동 생성 안 됨**: Agent가 슬라이드를 만들어도 프레젠테이션 제목은 "제목 없는 프레젠테이션" 유지
- **채팅 루프 흔적**: 드라이브 화면에서 "노란 디자인으로 수정해주세요"가 여러 번 반복된 채팅 기록이 있음 — Agent가 요청을 이행했는지 사용자가 판단할 피드백이 부족하다는 증거

#### 3-3. 설계 구조적 문제
- **HTML mode와 JSON mode 이중 렌더러 공존**: `html_content` null 체크로 분기. 두 모드 사이 편집 경험이 완전히 다름. 사용자 입장에서 예측 불가능
- **Agent 선택 UI 진입점이 숨겨짐**: AgentManagerPanel은 RightPanel 내 showManager state로 열리는데, 이를 여는 버튼이 현재 UI에서 제거된 상태. 에이전트 관리 페이지(`/agents`)를 따로 가야 함
- **파이프라인 선택 드롭다운이 채팅 헤더에 위치**: "기본 파이프라인"이 핵심 기능인데 채팅 입력 위에 조용히 숨어있음. 사용자가 이 개념을 이해할 진입점이 없음
- **세션(SessionSelector) UI 완전히 숨겨짐**: 코드에는 존재하지만 에디터에서 노출되지 않음

## 4. AI 기능 비교

### Snapdeck
- 단일 Agent(Snapdeck 1.0)이나 일관된 경험
- @컨텍스트 추가 지원
- 슬라이드별 재생성 가능
- Web Research, Web Scrap, Import Notion으로 외부 데이터 직접 주입

### Slidant
- 다중 Agent, 커스텀 파이프라인, 역할 분리(Content/Design/Layout) — **이론상 Snapdeck보다 강력**
- 그러나 사용자가 이 가치를 느끼는 시나리오가 설계되어 있지 않음
- 충돌 감지(ConflictResolver) 구현되어 있으나, 다중 Agent가 동시에 같은 슬라이드를 편집하는 워크플로우가 UI에서 유도되지 않음
- Agent 실행 중 단계 시각화(StepsChecklist)는 좋음 — 유지

## 5. 디자인 / 비주얼 품질

### Snapdeck
- 크림색 배경, 세리프 타이포, 얇은 보더 — 고급스러운 단색 톤
- 슬라이드 자체도 어두운 배경에 흰 텍스트로 전문적으로 보임
- UI와 콘텐츠 영역 색상 대비가 명확

### Slidant
- 흰 배경, 퍼플 어센트, 깔끔한 sans-serif — 기능적으로 나쁘지 않음
- 그러나 슬라이드 캔버스와 UI 패널의 톤이 분리되어 있지 않아 시각적 위계 불명확
- TopBar가 지나치게 미니멀(h-12, 아이콘 1개). 핵심 액션(export, share)이 없으니 당연한 결과지만 빈약해 보임

## 6. 퍼포먼스 / 안정성

- **HTML iframe 렌더링**: Slidant가 선택한 방향(CSS 전체 표현력)은 맞지만, sandbox="allow-same-origin"만 허용해 JS 없는 정적 HTML로 제한됨. 애니메이션 등 동적 요소 지원 안 됨
- **Snapdeck**: 동일하게 HTML 기반이나 콘텐츠 퀄리티가 일정함 — LLM 프롬프트 튜닝이 더 성숙함을 시사

## 7. 우선순위별 개선 목록

### P0 — 즉시 수정 (제품 최소 요건)

1. **Export PDF 구현** — jsPDF + html2canvas 또는 백엔드 puppeteer로 iframe HTML → PDF
2. **HTML 슬라이드 썸네일 생성** — Agent 완료 후 puppeteer/playwright로 슬라이드 PNG 캡처, `slides.thumbnail_url` 저장
3. **프레젠테이션 제목 자동 설정** — Agent 완료 응답의 summary에서 제목 추출해 자동 저장
4. **HTML mode 캔버스 직접 편집** — iframe 대신 shadow DOM 또는 srcdoc+postMessage 방식으로 요소 선택·이동 가능하게 전환. 현재는 Agent 결과물을 손댈 수 없음

### P1 — 단기 (차별점 살리기)

5. **다중 Agent 동시 실행 워크플로우 UX** — "디자인 Agent + 콘텐츠 Agent 동시 실행" 버튼. 지금은 기능이 있어도 사용자가 모름
6. **슬라이드 재생성 버튼** — 각 슬라이드 썸네일에 "이 슬라이드 다시 생성" 버튼
7. **프레젠테이션 모드** — 전체화면 발표 뷰어
8. **Undo/Redo** — HTML 모드에서도 `slide_history` 기반 즉시 롤백 버튼

### P2 — 중기 (격차 해소)

9. **공유 링크** — 읽기 전용 프레젠테이션 공개 URL
10. **외부 데이터 소스** — URL 스크랩, Notion import 등 Agent 입력 확장
11. **Agent 챗봇 온보딩** — 첫 사용자가 파이프라인 개념을 이해할 인터랙티브 가이드

## 8. 실사용 동작 비교 (동일 주제: "AI 슬라이드 툴 시장 분석 5장")

### 생성 플로우

| 단계 | Snapdeck | Slidant |
|------|----------|---------|
| 1 | 텍스트 입력 (랜딩 페이지) | 에디터 진입 후 채팅 입력 |
| 2 | **스타일 선택** (Report/Standard/Presentation/Keynote) | 없음 (바로 생성) |
| 3 | **테마 선택** (13종: Light/Dark/Neon/...) | 없음 |
| 4 | **레이아웃 변형 선택** (A/B/C/D/E 5종) | 없음 |
| 5 | **아웃라인 미리보기 + 편집** (차트 플레이스홀더 포함) | 없음 |
| 6 | Generate 클릭 → 슬라이드 5장 병렬 렌더링 (~9초) | 파이프라인 실행 (~35초) |
| 7 | 완료 | 완료 |

### 슬라이드 품질 비교 (동일 주제)

| 슬라이드 | Snapdeck 결과 | Slidant 결과 |
|----------|---------------|-------------|
| 표지 | 제목 + [발표자명] 플레이스홀더 + 장식 요소 | 제목 + 부제목 + 날짜 — 깔끔 |
| 목차 | 5항목 세로 목록 + 번호 뱃지 | INDEX + 01~04 번호 아이콘 — 둘 다 유사 |
| 시장현황 | **실제 라인차트** (2023~2026 수치 데이터 포함) | 불릿 + **"이미지 영역" 플레이스홀더** + CAGR 텍스트만 |
| 플레이어 비교 | **데이터 테이블** (행: 접근방식/강점/차별화, 열: 3개 툴) | 3개 카드 — 레이아웃 다양성 떨어짐 |
| 마지막 슬라이드 | KPI Cards + 결론 | 이모지 카드 3개 + 핵심 메시지 |

**핵심 격차**: Snapdeck은 아웃라인 단계에서 `[Line Chart]`, `[Data Table]`, `[2x2 Matrix]`, `[KPI Cards]` 타입을 명시하고 실제로 렌더링한다. Slidant는 LLM이 차트를 "이미지 영역"으로 처리해 플레이스홀더로 남긴다. 데이터 시각화 능력 차이가 결과물의 전문성을 가른다.

### 생성 중 UX

| 항목 | Snapdeck | Slidant |
|------|----------|---------|
| 생성 중 피드백 | 오른쪽 채팅 "Generating draft → Building slide structure..." | 파이프라인 단계별 체크리스트 (계획 수립→콘텐츠 기획→디자인 확정→슬라이드 생성) |
| 슬라이드 나타나는 방식 | 병렬 — 완성된 것부터 하나씩 팝인 | 전부 완료 후 한꺼번에 반영 |
| 캔버스 중간 상태 | 빈 슬라이드들이 그리드로 표시 (로딩 아이콘) | 기존 빈 슬라이드 유지, 완료 후 전환 |
| **승자** | 시각적 피드백 더 풍부 | 단계 이름이 구체적이어서 투명성 높음 |

**Slidant의 파이프라인 체크리스트는 실제로 좋은 UX다** — "계획 수립 → 콘텐츠 기획 → 디자인 확정 → 슬라이드 5개 병렬 생성"이 시각적으로 표현됨. 그러나 완료 전까지 캔버스가 빈 슬라이드로 유지되는 점은 답답함.

### 생성 후 수정 UX

| 항목 | Snapdeck | Slidant |
|------|----------|---------|
| 슬라이드별 재생성 | 썸네일 hover → 재생성 버튼 → 인라인 모달 프롬프트 | 채팅에 "@슬라이드N 수정..." 타이핑 |
| 요소 클릭 편집 | 캔버스 직접 클릭 → 텍스트/이미지 편집 | HTML mode에서 **불가** (iframe sandbox) |
| Undo | 버튼 있음 | 없음 (히스토리 패널만) |

## 결론

Slidant의 기술 기반(LangGraph 파이프라인, 다중 Agent, HTML 직렬화, 충돌 감지)은 Snapdeck보다 야심차다.  
그러나 현재 상태는 **엔지니어링 프로토타입**이지 **제품**이 아니다.

가장 위험한 것은 기술 과잉 투자와 UX 투자 부재의 불균형이다.  
Agent가 만든 슬라이드를 내보낼 수 없고, 직접 수정할 수 없고, 공유할 수 없다면  
다중 Agent 협업 아키텍처는 사용자에게 의미가 없다.

**핵심 명제**: Export → Thumbnail → 직접 편집 이 세 가지를 먼저 해결하고,  
그 다음에 다중 Agent 워크플로우를 사용자에게 보여줘라.
