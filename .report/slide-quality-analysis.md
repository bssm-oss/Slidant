## 2. 핵심 문제점

### 2-1. 레이아웃 단조로움 (Critical)

전체 10장이 **"제목 + 수평선 + 텍스트"** 패턴 반복.  
Snapdeck은 슬라이드 유형별로 카드 그리드, 타임라인, 계층 다이어그램, 콜아웃 블록 등을 혼합한다.

```
현재 구조 (10장 전부 동일):
┌─────────────┐
│ 제목         │
│ ──────────── │
│ • 텍스트     │
│ • 텍스트     │
│ • 텍스트     │
└─────────────┘

이상적 구조 (슬라이드 유형 분화):
타입A: 2-3개 카드 그리드 (구성 요소 설명)
타입B: 계층 다이어그램 (아키텍처)
타입C: 타임라인 (역사/배포 흐름)
타입D: 비교 테이블 (장점 vs 한계)
```

### 2-2. 시각적 요소 전무 (Critical)

기술 주제인데 다이어그램이 없다. 아래는 시각화가 필수인 슬라이드들:

| 슬라이드 | 필요한 시각 요소 | 현재 |
|----------|-----------------|------|
| Ceph 아키텍처 (4) | RADOS 레이어 블록 다이어그램 | 텍스트만 |
| 주요 구성 요소 (5) | OSD/MON/MDS/RGW 아이콘 카드 | 텍스트만 |
| CRUSH 알고리즘 (6) | 해시 → OSD 매핑 플로우 | 텍스트만 |
| 스토리지 인터페이스 (7) | 3-레이어 스택 다이어그램 | 텍스트만 |
| 실제 사용 사례 (9) | 기업 로고 + 아이콘 | 화살표 불릿만 |

### 2-3. 미완성 플레이스홀더 (Bug)

- **슬라이드 3**: 우측 절반이 빈 박스 ("Ceph 아키텍처" 텍스트만)
- **슬라이드 5**: 하단에 "Ceph Architecture — 주요 구성 요소" 레이블만 — 실제 다이어그램 없음
- **슬라이드 7**: 우측 "+" 아이콘 박스 — 내용 미삽입

### 2-4. 텍스트 오버플로우 (Bug)

슬라이드 4(Ceph 아키텍처)에서 마지막 항목 "CRUSH 알고리즘 — 중앙 위치 서버 없이 데이터 배치 결정"이 슬라이드 하단 경계를 벗어남.

### 2-5. 정보 밀도 과잉

슬라이드당 평균 5~6개 불릿. 청중 관점에서는 2~3개 핵심 포인트가 효과적.

```
현재: 슬라이드 6장 중 5개 항목 × 긴 설명문 = 읽을 내용이 너무 많음
목표: 핵심 키워드 1~2개 + 간결 설명 1줄
```

### 2-6. 배경/색상 단조로움

10장 전부 동일한 다크 네이비 배경. 섹션 전환이 시각적으로 구분되지 않는다.

## 3. Snapdeck 대비 품질 격차 요약

| 항목 | Slidant 현재 | Snapdeck 수준 |
|------|------------|--------------|
| 레이아웃 다양성 | 1종 (텍스트 목록) | 5~8종 (카드, 타임라인, 다이어그램 등) |
| 아이콘/SVG 사용 | 2~3개 (슬라이드 10만) | 슬라이드당 3~5개 |

| 플레이스홀더 미완성 | 3장 | 없음 |
| 텍스트 오버플로우 | 1장 | 없음 |

## 4. 개선 제안

### 단기 (프롬프트 개선으로 즉시 적용 가능)

**① 슬라이드당 텍스트 3개 이하 규칙**  
`HTML_COMPOSER_PROMPT`에 "한 슬라이드에 핵심 포인트 최대 3개, 각 설명 20자 이내" 명시.

**② 오버플로우 방지**  
슬라이드 높이 540px 기준으로 텍스트 블록이 넘치는지 `html_validator`에서 DOM 높이 체크 추가.

**③ 플레이스홀더 금지**  
프롬프트에 "빈 박스나 미완성 플레이스홀더 절대 금지. 삽입할 내용이 없으면 해당 영역을 제거하라" 명시.

---

### 중기 (HTML 생성 품질 향상)

**④ 아키텍처 슬라이드 → SVG 다이어그램 강제화**  
LLM에게 "계층 구조, 플로우, 관계도가 필요한 슬라이드는 텍스트 목록 대신 SVG/인라인 다이어그램으로 표현" 지시.  
예시 프롬프트 구절:
```
For slides describing system architecture or component relationships,
ALWAYS use an SVG diagram instead of a bullet list.
Use <svg> elements with rect, line, and text to draw layered block diagrams.
```

**⑤ 레이아웃 템플릿 다양화**  
프롬프트에 슬라이드 유형별 HTML 스니펫 예시를 제공:
- `card-grid`: 3~4개 아이콘 카드 (구성 요소 설명에 사용)
- `split-diagram`: 좌 텍스트 + 우 SVG (아키텍처에 사용)
- `timeline`: 수직/수평 타임라인 (역사, 배포 단계에 사용)
- `compare-table`: 2열 비교 (장점/단계, Before/After)
- `stat-highlight`: 숫자/지표 강조 (대규모 사례, 성능 수치)

**⑥ 아이콘 라이브러리 연동**  
인라인 SVG 아이콘 세트를 프롬프트에 제공하거나, 오픈소스 아이콘(예: Heroicons, Phosphor) CDN을 허용하여 LLM이 `<img>`나 `<svg use>`로 참조하게 한다.

**⑦ 섹션별 배경 컬러 시스템**  
타이틀 슬라이드는 진한 배경, 섹션 구분 슬라이드는 중간 색조, 내용 슬라이드는 어두운 배경으로 자동 분화하도록 프롬프트에 규칙 추가.

---

### 장기 (에이전트 시스템 개선)

**⑧ DesignAgent 활성화**  
현재 ContentAgent만 사용하는 것으로 보임. DesignAgent가 HTML 생성 후 시각적 검토 + 레이아웃 개선을 별도 패스로 수행하도록 파이프라인 구성.

**⑨ 슬라이드 유형 자동 분류**  
Planner 단계에서 각 슬라이드 유형(`type: "architecture"|"components"|"timeline"|"comparison"|"title"`)을 결정하고, Composer가 유형별로 다른 HTML 템플릿을 선택.

**⑩ html_validator에 시각 요소 체크 추가**  
현재 validator는 HTML 비어있음 여부만 체크. 아래 조건 추가:
- 아이콘/SVG 요소가 최소 1개 있는가
- 텍스트 노드 글자 수 < 300자인가
- 텍스트 overflow 없는가 (offsetHeight > scrollHeight 체크)
